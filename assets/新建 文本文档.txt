  二、通用 运行.bat 设计方案

  你需要的不是修修补补，而是一套项目类型自适应的自动化启动引擎。以下从检测层、环境层、构建层、运行层四个维度设计。

  1. 项目类型自动检测

  运行.bat 启动后，不要假设项目类型，而是通过特征文件来判定：

  ┌───────────────────────────────────┬────────────────────────────────────┐
  │         检测到的特征文件          │            推断项目类型            │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ pom.xml + src/main/java           │ Spring Boot / Java Maven 项目      │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ build.gradle                      │ Gradle 项目（Java / Kotlin）       │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ package.json + next.config.js     │ Next.js 项目                       │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ package.json + src/App.js         │ React SPA 项目                     │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ package.json + vue.config.js      │ Vue 项目                           │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ package.json + nuxt.config.js     │ Nuxt.js 项目                       │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ requirements.txt + manage.py      │ Django 项目                        │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ pyproject.toml + app.py           │ FastAPI / Flask 项目               │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ go.mod                            │ Go 项目                            │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ composer.json                     │ Laravel / PHP 项目                 │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ 同时有 backend/ + frontend/       │ 前后端分离项目（需递归检测子目录） │
  ├───────────────────────────────────┼────────────────────────────────────┤
  │ docker-compose.yml 但不含源码特征 │ 纯 Docker 编排项目                 │
  └───────────────────────────────────┴────────────────────────────────────┘

  检测策略：优先判断是否前后端分离（看根目录有无 backend/frontend 子目录），否则用特征文件从具体到通用的顺序匹配。

  2. 环境自动安装层

  核心思想：不要把环境安装捆绑在 运行.bat 里，而是拆成独立的 setup_env.bat + 缓存标记。
  每次运行只检查标记，没安装才触发安装。

  需要支持的运行时及安装策略：

  ┌────────────┬───────────────┬───────────────────────────────────────────────────────────────────────────────────┐
  │   运行时   │   检测方式    │                                   自动安装方案                                    │
  ├────────────┼───────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Java 17+   │ java -version │ 检测失败 → 从 Adoptium API 下载 便携版 JDK（不用系统安装，直接解压到项目 .jdk/    │
  │            │               │ 目录，配置 JAVA_HOME 和 PATH）                                                    │
  ├────────────┼───────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Node.js    │ node          │ 检测失败 → 从 nodejs.org 下载 便携版 到 .node/ 目录，或用 nvm-windows 自动安装    │
  │ 18+        │ --version     │                                                                                   │
  ├────────────┼───────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Python     │ python        │ 检测失败 → 下载嵌入版 Python 到 .python/ 目录                                     │
  │ 3.9+       │ --version     │                                                                                   │
  ├────────────┼───────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Maven      │ mvn --version │ 检测失败 → 下载 Maven 到 .maven/ 目录（或直接用 Gradle wrapper）                  │
  ├────────────┼───────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ pnpm /     │ 命令检测      │ 检测失败 → npm install -g pnpm（前提是 Node 已装好）                              │
  │ yarn       │               │                                                                                   │
  ├────────────┼───────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Docker     │ docker        │ 只提示，不自动安装（Docker Desktop 安装需要管理员且重启系统）                     │
  │            │ --version     │                                                                                   │
  ├────────────┼───────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ MySQL /    │ 端口检测      │ 优先检测 Docker → docker-compose up -d mysql redis；无 Docker 时提示手动安装      │
  │ Redis      │               │                                                                                   │
  └────────────┴───────────────┴───────────────────────────────────────────────────────────────────────────────────┘

  关键设计原则：
  - 便携版优先：JDK、Node、Python、Maven 都下载独立版本到项目目录下的隐藏文件夹，不污染系统环境
  - 缓存标记：安装完成后在 .bat_cache/ 写入标记文件，下次启动跳过检测
  - 版本锁定：在项目根目录的 .env 或 version.json 中写明所需版本号，而不是硬编码在 bat 里

  3. 构建执行层

  根据检测到的项目类型，执行对应的构建策略：

  [通用流程]
  1. 安装/验证运行时环境
  2. 读取项目配置文件，确定构建命令
  3. 执行构建
  4. 启动服务

  Spring Boot 后端

  - 策略：先找 mvnw.cmd（Maven Wrapper）→ 没有则用 mvn → 再没有用便携版 Maven
  - 构建：mvn clean package -DskipTests（注意原始脚本里的 -DskipDependencies 是错误的）
  - 启动：mvn spring-boot:run 或 java -jar target/*.jar

  Node.js / React / Next.js / Vue

  - 策略：检测 package.json 中 scripts 字段，优先 dev → start → serve
  - 包管理器：pnpm > yarn > npm（检测可用性）
  - 安装：pnpm install --frozen-lockfile（有 lockfile 时）或 pnpm install
  - 启动：pnpm dev 或读取 scripts 中第一个非 build 命令

  Python 项目（Django / FastAPI）

  - 策略：检测 Pipfile → poetry.lock → requirements.txt
  - 安装：pip install -r requirements.txt 或 poetry install
  - 启动：python manage.py runserver / uvicorn main:app

  前后端分离项目（当前项目类型）

  - 策略：先启动后端，再启动前端；如果检测到 docker-compose.yml，询问用户是否优先用 Docker
  - 端口管理：启动前用 netstat 检测端口占用，端口被占时自动递增

  Next.js 全栈项目

  - 策略：单命令 pnpm dev，但需检测 next.config.js 是否配置了自定义服务器

  4. 运行架构设计

  运行.bat（极简入口）
      │
      ├── 检查是否以管理员运行（记录权限级别）
      ├── 调用 lib/detect_project.bat → 输出 PROJECT_TYPE + CONFIG
      ├── 调用 lib/setup_env.bat → 确保所有运行时可用
      ├── 调用 lib/check_ports.bat → 扫描所需端口
      ├── 调用 lib/start_services.bat → Docker 启动基础设施
      ├── 调用 lib/build_and_run.bat → 按 PROJECT_TYPE 执行构建+启动
      └── 调用 lib/monitor.bat → 监视子进程，优雅退出

  每个 lib/*.bat 是独立模块，可复用、可单独测试。

  6. 适配"所有类型"项目的策略

  要做到真正的通用，需要引入项目描述文件的概念：

  在项目根目录放一个 .project.json（由生成项目的 AI 工具自动生成），内容如：

  {
    "name": "二手交易平台",
    "type": "spring-boot+react",
    "runtimes": {
      "java": "17",
      "node": "18"
    },
    "services": ["mysql:8.0", "redis:7.0", "elasticsearch:8.0"],
    "backend": {
      "path": "./backend",
      "build": "mvn clean package -DskipTests",
      "run": "mvn spring-boot:run",
      "port": 8080
    },
    "frontend": {
      "path": "./frontend",
      "packageManager": "pnpm",
      "run": "pnpm dev",
      "port": 5173
    }
  }

  这样 运行.bat 只需要读这个 JSON 文件（Windows 下用 cscript 调用 JScript 解析，或者自带一个极简 json.bat
  解析器），就可以完全通用——每类项目只需要不同的 .project.json 配置，不需要改 bat 本体。

  总结

  你的项目代码骨架是完整的，但缺 React 入口文件（App.js/index.js）和 Docker 基础设施配置。运行.bat
  目前的硬编码方案不可复用，改造为检测驱动 + 便携环境 + 模块化 lib + .project.json 描述文件的架构即可适配 Spring
  Boot、前后端分离、Next.js 等任意项目类型。核心思路是：bat 只做调度，项目特征由文件系统推断或 .project.json
  声明，环境安装走便携版下载。
